# 下載Adobe Analytics實施行動手冊

在開始之前， [下載行動手冊](assets/aa-implementation-playbook.xlsx)。

## 「業務需求」頁籤

**什麼：** 《商業需求文檔》（通常稱為BRD）是重要利益相關方、商業用戶和技術用戶希望進行協作的非常重要的文檔。 它是記錄您所需的所有KPI、報告要求以及您希望在Adobe Analytics(AA)實施完成時看到的任何資料點的位置。

**原因：** 這是後續文檔（SDR、技術規格等）的跳點 是AA協定終止狀態的共同真相來源。 本文檔將組織內各團隊的想法組織起來，形成一個指導方向，以推進實施工作的發展。

**方式：** 記錄業務需求通常由AA的最終業務用戶完成，但從技術用戶那裡獲得反饋非常重要，因為可能需要注意一些技術難題，而某些資料點可能比其他資料點需要付出更多努力，這就是確定優先順序的因素。

問自己：「我們希望在站點上跟蹤哪些內容」、「哪些資料點在報告使用中對我來說非常重要」，最重要的是，「這些資料點將如何為決策提供資訊」。 確保您的每項業務要求都與一個資料點相關，這一點非常重要，該資料點可用於為業務決策提供資訊。 例如，您很想跟蹤您站點上的每一點擊，但是，在一天結束時，您從報告中學到了什麼見解？

首先，在下面的螢幕抓圖（業務要求）中填寫C列。 這應該是「在我們的站點上完成了多少內部搜索」或「在印象方面哪個內部活動點最有效」之類的內容。 在填寫此詳細資訊級別後，您可以返回並填寫B列（類別），並將要求分組為「搜索」或「內部促銷」等類別，這些類別應與技術規格部分保持良好的對應。

您還將指出，使用eVar、事件、prop或組合是否可以實現您希望跟蹤的內容。

最後，在您開始向站點添加內容時，「實施狀態」列將用作狀態檢查。

## 變數映射頁籤（標籤文檔/SDR）

**什麼：** 標籤文檔（通常稱為SDR）是對AA的技術用戶和業務用戶都有價值的重要文檔。 它列出了報告套件使用的每個變數以及變數設定的所有相關詳細資訊、變數的實現方式及其在報告中的用途。 與您的屬性文檔一樣，這應該是一個活生生、管理良好的Excel文檔，由一個負責在引入標籤增強功能或實施更改時使其保持最新的點人負責。

**原因：** 本文檔將用於多種目的，但最重要的是：

* 對於任何新入職人員（新員工、希望更好地瞭解可用報告的業務負責人等） 本文檔提供了所有已實現變數的最佳視圖，以及它們的目的是什麼，以便個人能夠在學習AA設定時自我服務。
* 對於AA產品所有者/技術用戶，本文檔將提醒您如何設定其他變數，以及添加新維時可以使用哪些變數。

**方式：** 首先列出Excel文檔中所有Adobe的開箱變數（頁面、產品、地理等）以及eVars、道具、事件和清單變數。 每個站點/報告套件應有一個頁籤。
對於這些維，我添加以下列：
* **名稱：** 提供一個簡單、短名稱，大多數人都能理解。 這應該足夠直觀，新用戶可以拾取並瞭解變數要捕獲的內容。
* **描述：** 更詳細地介紹變數的用途和跟蹤的資料。 我保持這個簡短、簡單，並讓它與介面中使用的描述相匹配。 理想情況下，我不希望我的用戶需要咨詢標籤文檔。 因此，當在管理員後端設定新維時，我會在此處添加相同的說明。 這樣，用戶可以直接在工作區中按下資訊表徵圖，以瞭解維是什麼 — 無需拉出Excel文檔！

![頁面URL簡化](assets/page-url-simplified.png)

* **代碼：** 設定值的後端的代碼。 這可以是頁面上資料層中的欄位，也可以調出這是使用啟動規則、處理規則等完成的。
* **分類報告：** 調用使用分類導入程式或分類規則生成器執行的任何分類報告
* **解決方案範圍：** 我發現，在小列中列出所有屬性（至少是使用多於標準變數的屬性），並為該屬性上設定的每個維添加複選標籤非常有用。 這樣，您就可以輕鬆篩選特定屬性，並快速查看特定維的設定位置。
* **配置：** 每個變數的管理員UI設定（例如，eVars的過期、分配、促銷等）

![示例SDR](assets/sample-sdr.png)

還建議使用此標籤文檔跟蹤任何自由變數和任何「垃圾」變數。 當維不再有用時，開發人員通常需要一段時間才能將其刪除。 即使在此之後，也可能會發生快取，或者您可能會意識到，維也在其他位置設定。 清理尺寸並不容易，而且通常需要耐心。 以下是一些小貼士，可將垃圾郵件隱藏在床下，這樣用戶在跟蹤垃圾郵件時就不會感到困惑。

* 所有未使用的維/事件均為「free」或「bein deleted」
   * 如果維在過去90天內有垃圾值，則該維正在「刪除」
   * 如果維度至少在過去90天內是自由且清晰的，則它是「自由」
   * 在標籤文檔的「名稱」下將這些標籤標籤標籤為此類，以便您可以輕鬆篩選它們。 我在標籤文檔（Excel資料篩選器）中保持未選中這些選項，以便用戶看不到它們
   * 將這些標籤為介面中的eVar名，以便用戶在搜索中找不到它們(即「(v6)」)，並刪除介面中的說明
* 通過執行此操作，當需要新維時，可以在「名稱」列中輕鬆篩選「free」，以查找要使用的乾淨維
* 對於「正在刪除」的維和事件，我建議您使用Workspace跟蹤這些：
   * 建立僅對管理員可見的項目，包含3個表：Vars，道具和活動。 例如，我將「instances」用於特定的eVars，並將「prop5 exists」建立為HIT段的道具。
   * 將日期設定為最後90天
   * 將上面的內容用作3個表中的行以及具體值
   * 只要任何內容都到達「0」，我就會在標籤文檔中將其標籤為「free」，並將其從Workspace項目中刪除

這樣你的資料就永遠是乾淨的，你對垃圾有清楚的認識。

## 屬性頁籤

**什麼：** 屬性文檔應列出您的所有數字屬性 — 網站、移動應用、其他工具（聊天、反饋等），無論這些屬性是否用Adobe Analytics標籤。 這應作為跨業務和技術用戶的集中、活的文檔。

**原因：** 這樣，您就可以清楚地瞭解用戶在所有數字屬性上的旅程，以及Adobe Analytics所做的和未涵蓋的內容，以便您可以開始優先將標籤添加到任何缺少的屬性。 通過以此方式佈局您的數字生態系統，您可以在標籤策略中發現潛在的機會，以全面瞭解用戶的旅程。 例如 — 是否需要一個全局報告套件來跨多個域/站點進行跟蹤？ 域或應用之間是否需要訪問者ID切換以混合體驗？ 是否需要更新內部URL篩選器以進行跨域跟蹤？

**方式：** 確定文檔的所有者以提供治理和管理更新的單一責任來源。
在「屬性」頁籤上列出以下內容：
* **屬性名稱：** 這可以是域、子域、應用程式名稱等。 即使在同一域內，如果其某些部分是單獨管理的（如由不同的團隊或不同的技術管理），則應將這些部分分開。
* **連結(URL)** 到可用的屬性
* **所有者和聯繫人：** 列出屬性的主所有者或聯繫人
* **標籤方法：** 我們中的許多人都有不同的代碼方法和實施（啟動、JS檔案、AEP等）。 如果需要，您可以進一步細分此內容（如代碼版本或標籤管理系統），但這樣做是為了跟蹤所有不同的代碼方法和版本、代碼需要更新的位置以及需要如何維護。 如果使用Adobe啟動，請列出啟動屬性名稱。

請記住包括所有數字屬性，即使這些屬性沒有用Adobe Analytics標籤。 這將幫助您瞭解您的數字環境以及用戶如何與您的所有屬性交互。

建議盡可能簡單地保存本文檔，並且不要用過多的資訊將其阻塞，以便組織中不同部門仍然能夠輕鬆理解。 分析團隊通常比其他團隊更瞭解數字環境，因此其他團隊和執行官經常使用此文檔來提供全面的概述。

>[!TIP]
>
>在Adobe Analytics建立站點名稱/屬性維。 在Adobe Analytics有一個專用維(通常是eVar)，該維標識站點名稱/應用名稱，將允許分段、故障排除、虛擬報告套件建立等。 好處無窮無盡，特別是當將多個站點合併到一個（全球）報告套件中時。 關鍵是確保您的開發人員團隊始終在屬性維中設定此值，包括所有頁載入(s.t calls/trackState)和所有自定義事件(s.tl calls/trackAction)。 處理規則可能是幫助您正確且一致地設定這些值的寶貴工具。
